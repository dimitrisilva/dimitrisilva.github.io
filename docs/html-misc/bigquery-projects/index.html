Google BigQuery Project Guide

Introduction

Google BigQuery is a fully managed, serverless data warehouse that enables fast SQL queries over very large datasets. Unlike a traditional database server that you provision and maintain, BigQuery abstracts away all infrastructure management: you upload data, write SQL, and pay only for the bytes scanned by your queries. This makes it particularly well-suited for researchers and analysts who need to work with gigabytes or terabytes of data without the overhead of managing a server.

This guide walks through the end-to-end process of building a BigQuery project, from creating a Google Cloud account and configuring billing all the way through querying, transforming, exporting results, and accessing data programmatically from Python or R. Each step is self-contained, but they are designed to be followed in order for a first-time setup.

***

The workflow covered in this guide consists of five major stages. The first stage, project creation and billing setup, covers creating a Google Cloud project, configuring a billing account (including Pix payments for Brazilian residents), enabling the necessary APIs, and setting a budget alert to avoid unexpected charges. These are prerequisites for everything that follows.

The second stage, data upload and storage, covers creating a Google Cloud Storage (GCS) bucket and uploading raw data files (JSONL, CSV, and schema JSON files) from a local machine. GCS acts as an intermediate staging area: for large files it is more reliable and faster to load data into BigQuery from GCS than to stream it directly from a local machine.

The third stage, building and clustering tables in BigQuery, covers creating a BigQuery dataset (the container for tables), defining table schemas, and loading data with partitioning and clustering. Partitioning splits a table into segments by a date or integer-range column, while clustering physically sorts data within each partition by one or more columns. Together they dramatically reduce the amount of data scanned per query, which directly lowers cost and improves performance.

The fourth stage, data transformation and export, covers running SQL inside BigQuery to produce derived or analysis-ready tables and exporting the results back to GCS as JSONL files. BigQuery's EXPORT DATA statement bypasses the normal query-result size limits and can export petabyte-scale results into automatically sharded files, making it the correct tool for any non-trivial output.

The fifth and final stage, service accounts and external access, covers creating a service account (a non-human identity with specific permissions), generating a JSON key file, and using that key to authenticate from Python or R. This enables programmatic access to BigQuery data without exposing personal credentials.

***

A few conventions used throughout this guide. Shell commands are shown in pre blocks; on Windows, commands that use the caret ^ character as a line continuation are intended for Command Prompt or PowerShell. On macOS/Linux, replace ^ with \. Placeholder values such as [PROJECT_ID] or your_clustering_column should be substituted with values specific to your project before running a command. Tips, warnings, and notes are highlighted in coloured boxes throughout.

Step 1 — Project Creation and Billing Setup

This step establishes the infrastructure foundation: a Google Cloud project, a billing account, the required APIs, and a budget alert. None of the subsequent steps are possible without completing this one first.

1.1 Create a Google Cloud Project

A Google Cloud project is the top-level organisational unit for all resources, APIs, billing, and permissions. Every resource you create — a BigQuery dataset, a Cloud Storage bucket, a service account — belongs to exactly one project.

  1. Go to the Google Cloud Console and sign in with your Google account.
  2. In the top navigation bar, click the project selector dropdown and choose New Project.
  3. Give the project a name (e.g., us-datasets). The console will auto-generate a globally unique Project ID based on this name; you can customise it if needed.
  4. Leave the organisation field blank if you are working individually.
  5. Click Create.
Region selection: The project itself has no region, but resources created inside it do. For BigQuery datasets and Cloud Storage buckets, choose us-central1 (Iowa). This region is covered by Google Cloud's Always Free tier, which includes 10 GB of BigQuery storage and 1 TB of query processing per month at no cost.

1.2 Configure Billing with Pix (Manual Payments)

If you do not have an international credit card, you can fund your account using Pix, Brazil's instant payment system, by selecting Brazil as your billing country. This enables a prepaid (manual payments) mode in which you charge the account with a specific amount before running queries.

  1. Go to console.cloud.google.com/billing and click Create account.
  2. Set Country to Brazil. The currency will automatically be set to BRL (Brazilian Real).
  3. Under Payment type, select Manual Payments.
If only Automatic Payments is shown, try registering a domestic debit card first. You may be able to switch to manual payments afterwards from the billing settings page.

To add credit to the account via Pix:

  1. Navigate to Billing → Payment overview.
  2. Click Make a Payment.
  3. Select Pix as the payment method, generate a QR code, and complete the payment from your banking app.
Tip: Load a small amount (e.g., R$ 50) before starting work. Since the account operates as a prepaid wallet, running out of credit mid-session will halt all queries and uploads.

1.3 Enable Required APIs

Newly created Google Cloud projects have most APIs disabled by default. You must explicitly enable BigQuery and Cloud Storage before using them. There are two ways to do this.

Method A — Google Cloud Console (GUI)

  1. Open the left navigation menu (☰) and go to APIs & Services → Library.
  2. Search for BigQuery API, select it, and click Enable.
  3. Search for Cloud Storage (or Google Cloud Storage JSON API), select it, and click Enable.

Method B — Google Cloud SDK / CLI (Recommended)

If the Google Cloud SDK is installed, run the following commands from a terminal (Anaconda Prompt, PowerShell, or Bash):

# Set the active project (replace with your actual Project ID)
gcloud config set project [PROJECT_ID]

# Enable BigQuery and Cloud Storage APIs in one command
gcloud services enable bigquery.googleapis.com \
    storage-api.googleapis.com \
    storage-component.googleapis.com

To confirm that the APIs have been enabled correctly:

gcloud services list --enabled --filter="name:(bigquery* storage*)"

You should see entries for bigquery.googleapis.com, storage-api.googleapis.com, and storage-component.googleapis.com in the output.

APIWhy it is needed
bigquery.googleapis.com Executing SQL queries, creating datasets and tables, clustering and partitioning operations.
storage-api.googleapis.com Reading and writing objects (files) in Cloud Storage buckets, including staging data before loading into BigQuery.
storage-component.googleapis.com Low-level storage operations required internally by several GCS-dependent services.

1.4 Set a Budget Alert

Even with the Always Free tier, unexpected activity (e.g., an accidentally large query or an unclosed streaming insert) can generate charges. Setting a budget alert sends an email notification before you reach a spending threshold.

  1. Go to Billing → Budgets & alerts and click Create budget.
  2. Scope the budget to your project.
  3. Set a total amount (e.g., R$ 20) and enable email alerts at 50%, 90%, and 100% of the threshold.
  4. Click Finish.
Budget alerts do not automatically stop spending — they only notify you. To cap spending automatically, you would need to configure a billing account budget with spend-based notifications and a Cloud Function to disable billing, which is beyond the scope of this guide.

Step 2 — Data Upload and Storage Strategy

While BigQuery supports direct file uploads from a local machine for small files, the recommended approach for datasets of several gigabytes or more is to first transfer files to a Google Cloud Storage (GCS) bucket and then load from there into BigQuery. This approach is faster, more reliable for large files, and avoids timeout issues associated with direct uploads over slow connections.

2.1 Create a Cloud Storage Bucket

A GCS bucket is a flat container for objects (files). Buckets have globally unique names, and their region should match the BigQuery dataset region to avoid inter-region data transfer fees.

  1. In the Console, navigate to Cloud Storage → Buckets and click Create.
  2. Bucket name: Choose a globally unique name (e.g., us-datasets-analysis). Names must be lowercase, 3–63 characters, and contain only letters, numbers, and hyphens.
  3. Location type: Select Region and choose the same region as your BigQuery dataset (e.g., us-central1).
  4. Storage class: Select Standard for data that is actively read and written.
  5. Leave all other settings at their defaults and click Create.
Tip: If this bucket is only used as a staging area during a single analysis session, remember to delete it once you have finished to avoid ongoing storage charges.

2.2 Upload Data Files

Use the gsutil command-line tool (bundled with the Google Cloud SDK) to transfer files from your local machine to the bucket.

2.2.1 Verify the bucket exists

gsutil ls gs://us-datasets-analysis/

2.2.2 Upload data files

The -m flag enables parallel (multi-threaded) transfers, which significantly accelerates uploads of many files or large files. The -n flag skips files that already exist at the destination, making the command safe to re-run.

# Upload JSONL data files
gsutil -m cp -n "D:\input-folder\*.jsonl" gs://us-datasets-analysis/raw-data/

# Upload CSV data files
gsutil -m cp -n "D:\input-folder\*.csv" gs://us-datasets-analysis/raw-data/

# Upload schema definition files
gsutil -m cp -n "D:\input-folder\*.json" gs://us-datasets-analysis/schema/
Replace D:\input-folder\ with the actual path to your local data directory. On macOS/Linux, use Unix-style paths (e.g., /home/user/input-folder/*.jsonl).

2.2.3 Confirm the upload

gsutil ls gs://us-datasets-analysis/raw-data/

The output should list all uploaded files. If a file is missing, check the source path and re-run the upload command.

Step 3 — Building and Clustering Tables in BigQuery

This step covers creating the BigQuery dataset container, reviewing your schema files, and loading data into tables with partitioning and clustering applied. The choices you make here directly determine query performance and cost.

3.1 Create a BigQuery Dataset

In BigQuery, a dataset is a container for tables, analogous to a schema in a traditional relational database. A dataset belongs to a single region; data loaded into it from GCS must come from a bucket in the same region.

# Create a dataset named 'us_data' in the us-central1 region
bq --location=us-central1 mk -d us_data

To confirm creation:

bq ls -d

3.2 Prepare Schema Files

A BigQuery schema is a JSON array that describes the columns in a table — their names, data types, and modes (NULLABLE, REQUIRED, or REPEATED). Before loading data, verify that your schema files (stored at gs://us-datasets-analysis/schema/*.json) are correct. Pay particular attention to the following types:

  • INTEGER — used for numeric IDs and range-partitioning keys.
  • DATE / TIMESTAMP — used for time-based partitioning fields.
  • GEOGRAPHY — used for geospatial columns loaded from GeoJSON; requires the --json_extension=GEOJSON flag during load.

3.3 Load Data into Tables

BigQuery supports several optimisation strategies at load time. The three most common are described below.

Case A — Clustering only

Clustering physically sorts the data in storage by the values of one or more columns. Queries that filter on a clustered column skip blocks of data that cannot match, reducing bytes scanned and therefore cost.

bq load ^
  --source_format=NEWLINE_DELIMITED_JSON ^
  --json_extension=GEOJSON ^
  --clustering_fields=your_clustering_column ^
  --schema=D:\schema\schema_clustered_data.json ^
  us_data.clustered_table ^
  "gs://us-datasets-analysis/raw-data/*.jsonl"

Case B — Partitioning by date column

Partitioning splits a table into segments by the values of a date or timestamp column. A query that filters on the partition column only reads the relevant partitions rather than the entire table.

bq load ^
  --source_format=NEWLINE_DELIMITED_JSON ^
  --json_extension=GEOJSON ^
  --time_partitioning_field=your_datetime_column ^
  --schema=D:\schema\schema_partitioned_data.json ^
  us_data.partitioned_table ^
  "gs://us-datasets-analysis/raw-data/*.jsonl"

Case C — Partitioning by integer range

When no date column exists but data can be segmented by a numeric ID range, integer-range partitioning divides the table into equal-width buckets.

bq load ^
  --source_format=NEWLINE_DELIMITED_JSON ^
  --json_extension=GEOJSON ^
  --range_partitioning_field=id ^
  --range_partitioning_start=0 ^
  --range_partitioning_end=100000 ^
  --range_partitioning_interval=1000 ^
  --schema=D:\schema\schema_partitioned_data.json ^
  us_data.partitioned_table ^
  "gs://us-datasets-analysis/raw-data/*.jsonl"

3.4 Combining Partitioning and Clustering (Recommended)

Using partitioning and clustering together is the most cost-effective strategy for large tables that are queried by both a time dimension and a categorical or numeric dimension. BigQuery first partitions the data (e.g., by date), then sorts each partition by the clustering columns (e.g., an integer ID). A query that filters on both dimensions can skip entire partitions and irrelevant blocks within each partition.

Example for JSONL/GeoJSON files:

bq load \
  --source_format=NEWLINE_DELIMITED_JSON \
  --json_extension=GEOJSON \
  --time_partitioning_field=your_datetime_column \
  --clustering_fields=your_clustering_column \
  --schema=D:\schema\schema_data.json \
  us_data.optimized_table \
  "gs://us-datasets-analysis/raw-data/*.jsonl"

Example for CSV files:

bq load ^
  --source_format=CSV ^
  --skip_leading_rows=1 ^
  --time_partitioning_field=your_datetime_column ^
  --clustering_fields=your_clustering_column ^
  --schema=D:\schema\schema.json ^
  us_data.optimized_table ^
  "gs://us-datasets-analysis/raw-data/*.csv"

3.5 Verify the Table Structure

After loading, inspect the table to confirm that partitioning and clustering were applied as expected:

# List all tables in the dataset
bq ls us_data

# Show full metadata for a specific table (including partition and cluster info)
bq show us_data.optimized_table
StrategyBest suited forMechanism
Clustering Filters on categorical or numeric columns (e.g., region, ID) Physically sorts blocks within storage; query skips irrelevant blocks
Time partitioning Filters on date or timestamp columns Splits table into daily/monthly/yearly segments; query reads only matching partitions
Integer-range partitioning Filters on a numeric ID with a known range Splits table into equal-width numeric buckets; query reads only matching buckets
Partitioning + clustering Filters on both a time dimension and a categorical/numeric dimension Combines both mechanisms for maximum data-skip at query time

Step 4 — Data Transformation and Export

With data loaded into BigQuery, you can now run SQL to transform, join, or aggregate it, materialise the results as new tables, and export those tables back to Cloud Storage for downstream use (e.g., in Python with geopandas or in R with sf).

4.1 Create an Export Folder in GCS

GCS does not have true directories — all objects live in a flat namespace — but you can simulate a folder hierarchy by using path-like prefixes in object names (e.g., treated-data/county/). The simplest way to create an empty "folder" is to upload a placeholder object and immediately delete it:

echo . > empty.txt
gsutil cp empty.txt gs://us-datasets-analysis/treated-data/county/
gsutil rm gs://us-datasets-analysis/treated-data/county/empty.txt
In practice, running an EXPORT DATA statement with a GCS URI automatically creates the necessary prefix, so this step is optional. It is useful mainly for visual organisation in the Console.

4.2 Transform Data and Export Results as JSONL

The recommended workflow is to first create a derived table with CREATE OR REPLACE TABLE, then export it with EXPORT DATA. Run the following SQL in the BigQuery Console or via bq query:

-- 1. Create (or replace) the result table
CREATE OR REPLACE TABLE `us_data.county_map_meet`
AS
-- Optionally include a WITH clause for intermediate transformations
SELECT *
FROM `us_data.optimized_table`
WHERE your_filter_condition;

-- 2. Export the result table to GCS as JSONL
EXPORT DATA OPTIONS(
  uri='gs://us-datasets-analysis/treated-data/county/county_map_meet_1960-2020*.jsonl',
  format='JSON',
  overwrite=true
) AS
SELECT
  'Feature'               AS type,
  STRUCT(* EXCEPT(geometry)) AS properties,
  geometry
FROM `us_data.county_map_meet`;

The * wildcard in the output URI instructs BigQuery to automatically shard the output into multiple files when the result set exceeds a single file's size limit. This is essential for large exports.

GeoJSON compatibility: The EXPORT DATA statement serialises GEOGRAPHY columns as GeoJSON-compatible strings. The output JSONL files can be read directly in Python with geopandas.read_file() or converted to an sf object in R, without any additional post-processing.

4.3 Export the Table Schema

Saving the schema of a result table alongside the data is important for reproducibility: it allows you to reload or validate the data later without guessing column types. The following commands extract the schema locally and upload it to GCS:

# Extract schema to a local file
bq show --schema --format=prettyjson us_data.county_map_meet ^
  > D:\Downloads\schema_county_map_meet.json

# Upload the schema file to the GCS schema folder
gsutil -m cp -n D:\Downloads\schema_county_map_meet.json ^
  gs://us-datasets-analysis/schema/

4.4 Download Results and Clean Up

After exporting, download the result files to your local machine and remove intermediate cloud resources to avoid ongoing storage charges.

# Download exported JSONL files
gsutil -m cp ^
  "gs://us-datasets-analysis/treated-data/county/county_map_meet_1960-2020*.jsonl" ^
  D:\Downloads\
# Delete intermediate BigQuery tables
bq rm -f us_data.optimized_table
bq rm -f us_data.county_map_meet

# Delete raw data from GCS
gsutil rm "gs://us-datasets-analysis/raw-data/*.jsonl"
Warning: bq rm -f and gsutil rm are irreversible. Confirm that you have downloaded all files you need before running cleanup commands.

Key Notes for Step 4

  • Large result sets: The EXPORT DATA statement bypasses BigQuery's interactive query result size limits (typically capped at a few GB in the Console) and can export results of any size by sharding output across multiple files.
  • GeoJSON affinity: Exported JSONL files containing GEOGRAPHY columns are GeoJSON-compatible, making them immediately usable with spatial libraries such as Python's geopandas or R's sf.
  • Cost awareness: BigQuery charges for data scanned, not data stored (beyond the free tier). Exporting results does not incur additional query charges, but storing large files in GCS does accrue storage fees over time.

Step 5 — Service Accounts and External Access

To query BigQuery programmatically from Python or R, you need to authenticate your scripts against Google Cloud. The recommended approach is to create a service account — an identity associated with your project rather than your personal Google account — and use a JSON key file to authenticate.

5.1 Create a Service Account

A service account is a special type of Google account intended for non-human use. Scripts, pipelines, and applications authenticate as a service account rather than as a person.

  1. In the Console, navigate to IAM & Admin → Service Accounts.
  2. Click + Create Service Account.
  3. Enter a name (e.g., analysis-bot) and an optional description, then click Create and Continue.

5.2 Assign Roles

Grant the service account only the permissions it needs. The following five roles cover typical research workloads:

RolePurpose
BigQuery Job UserRequired to execute queries (jobs) against the project.
BigQuery Data EditorCreate, update, and delete BigQuery tables and datasets.
BigQuery Read Session UserRequired for high-throughput data reads from Python/R via the BigQuery Storage API.
Storage Object AdminCreate and delete objects (files) in Cloud Storage buckets.
Storage Object ViewerList and read objects in Cloud Storage buckets.

After adding each role, click Continue and then Done.

5.3 Generate and Download a JSON Key File

The JSON key file is the credential that your scripts will use. Treat it like a password.

  1. On the Service Accounts list page, click the email address of the account you just created.
  2. Go to the Keys tab and click Add Key → Create new key.
  3. Select JSON and click Create. A file is downloaded automatically.
  4. Move the file to a secure local directory, e.g., D:\Keys\google-auth.json.
Security: Never commit the JSON key file to a public repository (GitHub, GitLab, etc.). Add it to your .gitignore. If it is accidentally exposed, revoke it immediately from the Keys tab and generate a new one.

Python — Accessing BigQuery with google-cloud-bigquery

The official Python client library provides a convenient interface for running queries, loading data, and downloading results as Pandas DataFrames.

Installation

pip install google-cloud-bigquery pandas pyarrow

Connection template

The following functions provide a reusable template for authenticating and querying BigQuery from Python:

import os
from google.cloud import bigquery
import pandas as pd


def get_bq_client(json_path: str) -> bigquery.Client:
    """Initialise a BigQuery client using a service account JSON key."""
    os.environ["GOOGLE_APPLICATION_CREDENTIALS"] = json_path
    return bigquery.Client()


def query_to_df(sql: str, json_path: str) -> pd.DataFrame:
    """Run a SQL query and return the results as a Pandas DataFrame."""
    client = get_bq_client(json_path)
    query_job = client.query(sql)
    return query_job.to_dataframe()


# ── Usage ──────────────────────────────────────────────────────────────────
key_path = "D:/Keys/google-auth.json"

sql_query = """
    SELECT *
    FROM `us_data.county_map_meet`
    LIMIT 100
"""

df = query_to_df(sql_query, key_path)
print(df.head())
For geospatial work, consider using the google-cloud-bigquery-storage package together with geopandas. The BigQuery Storage API reads data in parallel using Arrow or Avro serialisation, which is substantially faster than the default REST-based download for large tables.

R — Accessing BigQuery with bigrquery

The bigrquery package provides a tidyverse-compatible interface to BigQuery. It integrates with dplyr, allowing you to write database queries using familiar R syntax.

Installation

install.packages("bigrquery")

Connection template

The following function authenticates with a service account JSON key and returns query results as a tibble:

library(bigrquery)
library(dplyr)


get_bq_data <- function(sql, project_id, json_path) {
  # Authenticate using the service account key
  bq_auth(path = json_path)

  # Execute the query and download results
  tb <- bq_project_query(project_id, sql)
  bq_table_download(tb)
}


# ── Usage ──────────────────────────────────────────────────────────────────
key_path    <- "D:/Keys/google-auth.json"
project_id  <- "your-project-id"   # The Project ID from Step 1.1

sql_query <- "
  SELECT *
  FROM `us_data.county_map_meet`
  LIMIT 100
"

df <- get_bq_data(sql_query, project_id, key_path)
head(df)
bigrquery also supports a dplyr-based interface via DBI and dbplyr. You can connect with DBI::dbConnect(bigrquery::bigquery(), project = project_id, dataset = "us_data") and then use tbl(), filter(), mutate(), and other dplyr verbs, which are translated to SQL and executed server-side.
